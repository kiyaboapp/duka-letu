# Requirements Document

## Introduction

Complete replacement of the broken Bootstrap-based UI in Kiyabo Duka (a Django 5.x shop management
ERP for Upendo Stationery, Dar es Salaam, TZS currency) with a Tailwind CSS standalone
implementation. All templates currently reference Bootstrap grid and component classes that are not
installed, and `{% load crispy_forms_tags %}` tags that are not installed. This overhaul installs
Tailwind CSS as a single pre-compiled static file (no Node.js, no npm), replaces all 25 templates
with a consistent dark-sidebar layout, adds Tailwind widget attrs to all three form files so Django
renders styled inputs natively, and removes the Bootstrap static files.

## Glossary

- **System**: The Kiyabo Duka Django application as a whole.
- **Base_Template**: `templates/base.html` — the single layout shell all page templates extend.
- **Template_Set**: The 25 HTML template files listed in the design document that must be replaced or created.
- **Form_Files**: The three Python form modules: `sales/forms.py`, `credit/forms.py`, `inventory/forms.py`.
- **Tailwind_CSS**: The pre-compiled `static/css/tailwind.css` file generated by the Tailwind CSS standalone CLI.
- **Sidebar**: The fixed dark navigation panel defined in Base_Template.
- **KPI_Card**: The summary metric card component defined in the design document.
- **Data_Table**: The list-view table component defined in the design document.
- **Badge**: The status pill component defined in the design document.
- **Button**: The action link/button component defined in the design document.
- **Page_Header**: The page title + action area component defined in the design document.
- **Flash_Message**: A Django `messages` framework message rendered in Base_Template.
- **Widget_Attrs**: The `attrs` dictionary on a Django form widget that injects HTML attributes.
- **Bootstrap_Class**: Any CSS class name from the Bootstrap framework, including but not limited to `col-md-*`, `d-flex`, `btn btn-*`, `table table-*`, `badge bg-*`, `row`, `container`.
- **PDF_Template**: `templates/reports/income_statement_pdf.html` — a standalone print-ready document that does not extend Base_Template.

## Requirements

### Requirement 1: Tailwind CSS Static Asset Setup

**User Story:** As a developer, I want a single pre-compiled Tailwind CSS file served as a Django
static asset, so that the application has a working CSS foundation with no Node.js build pipeline.

#### Acceptance Criteria

1. THE System SHALL serve Tailwind_CSS from `static/css/tailwind.css` via Django's static file
   mechanism.
2. THE System SHALL NOT include `static/css/bootstrap.min.css` in the static directory.
3. THE System SHALL NOT include `static/js/bootstrap.bundle.min.js` in the static directory.
4. THE Base_Template SHALL reference Tailwind_CSS using `{% load static %}` and
   `{% static 'css/tailwind.css' %}`.

---

### Requirement 2: Base Template — Dark Sidebar Layout

**User Story:** As a shop operator, I want a consistent dark sidebar navigation layout on every
page, so that I can navigate between all app sections without confusion.

#### Acceptance Criteria

1. THE Base_Template SHALL define template blocks named `title`, `content`, `extra_css`, and
   `extra_js`.
2. THE Base_Template SHALL render a fixed dark Sidebar using Tailwind classes (`bg-gray-900`,
   `w-64`, `flex-shrink-0`) containing navigation links to: Dashboard, Sales, Credit Sales,
   Inventory, Products, Finance, Assets, and Reports.
3. WHEN the current page's `request.resolver_match.app_name` matches a Sidebar navigation entry,
   THE Base_Template SHALL apply active state classes (`bg-blue-600 text-white`) to that entry and
   inactive state classes (`text-gray-300 hover:bg-gray-800`) to all other entries.
4. WHEN Django messages are present in the request context, THE Base_Template SHALL render each
   Flash_Message with a color class determined by its tag: `success` → green (`bg-green-50
   border-green-200 text-green-800`), `error` → red (`bg-red-50 border-red-200 text-red-800`),
   `warning` → yellow (`bg-yellow-50 border-yellow-200 text-yellow-800`), `info` → blue
   (`bg-blue-50 border-blue-200 text-blue-800`).
5. THE Base_Template SHALL NOT contain any Bootstrap_Class.
6. WHEN the viewport is below the `md` breakpoint, THE Sidebar SHALL be hidden by default and a
   hamburger button in the top header SHALL be visible.
7. WHEN the hamburger button is clicked, THE Base_Template SHALL toggle Sidebar visibility via
   inline JavaScript without requiring any external JS library.

---

### Requirement 3: Template Replacement — All 25 Templates

**User Story:** As a shop operator, I want every page in the application to render correctly using
Tailwind CSS, so that the UI is functional and consistent across all modules.

#### Acceptance Criteria

1. THE Template_Set SHALL consist of exactly these 25 files:
   `templates/base.html`,
   `templates/dashboard/index.html`,
   `templates/sales/index.html`,
   `templates/sales/sale_list.html`,
   `templates/sales/sale_form.html`,
   `templates/sales/return_inward_form.html`,
   `templates/inventory/index.html`,
   `templates/inventory/purchase_list.html`,
   `templates/inventory/purchase_form.html`,
   `templates/credit/index.html`,
   `templates/credit/debtor_list.html`,
   `templates/credit/debtor_detail.html`,
   `templates/credit/debtor_form.html`,
   `templates/credit/debt_form.html`,
   `templates/credit/debt_return_form.html`,
   `templates/finance/index.html`,
   `templates/assets/index.html`,
   `templates/catalog/index.html`,
   `templates/catalog/product_list.html`,
   `templates/catalog/product_detail.html`,
   `templates/reports/index.html`,
   `templates/reports/income_statement.html`,
   `templates/reports/income_statement_pdf.html`,
   `templates/reports/stock_report.html`,
   `templates/reports/debtor_aging.html`.
2. WHEN any template in Template_Set (except PDF_Template) is rendered, THE System SHALL produce
   HTML that extends Base_Template and defines a `{% block content %}` block.
3. THE Template_Set SHALL NOT contain any Bootstrap_Class in any template file.
4. THE Template_Set SHALL NOT contain any `{% load crispy_forms_tags %}` tag in any template file.
5. THE Template_Set SHALL NOT contain any inline `<style>` blocks in any template file.
6. WHEN a list view template is rendered with an empty queryset, THE Data_Table SHALL display an
   empty-state row with a descriptive message spanning all columns.
7. THE `templates/catalog/index.html` file SHALL be created as a new file (it does not currently
   exist as a Tailwind template).

---

### Requirement 4: Form Widget Styling — No crispy-forms

**User Story:** As a shop operator, I want all data-entry forms to render with styled inputs, so
that I can enter data comfortably without relying on the crispy-forms library.

#### Acceptance Criteria

1. THE Form_Files SHALL NOT import or reference `crispy_forms` anywhere.
2. THE SaleForm, ReturnInwardForm, SaleOfficeUseForm, and DrawingForm in `sales/forms.py` SHALL
   define Widget_Attrs on every field widget that include the Tailwind input class string
   `mt-1 block w-full rounded-md border-gray-300 shadow-sm focus:border-blue-500
   focus:ring-blue-500 sm:text-sm`.
3. THE DebtorForm, DebtForm, and DebtReturnForm in `credit/forms.py` SHALL define Widget_Attrs on
   every field widget that include the Tailwind input class string
   `mt-1 block w-full rounded-md border-gray-300 shadow-sm focus:border-blue-500
   focus:ring-blue-500 sm:text-sm`.
4. THE PurchaseForm, PurchaseDetailForm, and ReturnOutwardForm in `inventory/forms.py` SHALL define
   Widget_Attrs on every field widget that include the Tailwind input class string
   `mt-1 block w-full rounded-md border-gray-300 shadow-sm focus:border-blue-500
   focus:ring-blue-500 sm:text-sm`.
5. WHEN a form from Form_Files is rendered by Django, THE rendered HTML SHALL contain the Tailwind
   input class on each `<input>`, `<select>`, and `<textarea>` element produced by that form.
6. WHEN a form field has validation errors, THE form template SHALL render each error message in
   red text (`text-red-600`) below the field.

---

### Requirement 5: UI Component Consistency

**User Story:** As a shop operator, I want a visually consistent set of UI components across all
pages, so that the application feels professional and is easy to scan.

#### Acceptance Criteria

1. THE KPI_Card component SHALL display a circular icon area, a TZS-formatted numeric value using
   `tabular-nums`, and a descriptive label on every dashboard and report summary row that uses it.
2. THE Data_Table component SHALL use `divide-y divide-gray-200` on `<tbody>`, `hover:bg-gray-50`
   on data rows, and `tabular-nums` on numeric cells in every list view.
3. WHEN a Badge is rendered, THE Badge SHALL use exactly one of four color variants: green
   (`bg-green-100 text-green-800`) for success/in-stock, yellow (`bg-yellow-100 text-yellow-800`)
   for warning/low-stock, red (`bg-red-100 text-red-800`) for danger/overdue, or gray
   (`bg-gray-100 text-gray-700`) for neutral/pending states.
4. WHEN an action Button is rendered, THE Button SHALL use exactly one of the four defined variants:
   primary (`bg-blue-600 text-white hover:bg-blue-700`), secondary (border `border-gray-300
   text-gray-700 hover:bg-gray-50`), accent (`bg-emerald-600 text-white hover:bg-emerald-700`), or
   small-action (`bg-blue-50 text-blue-700 hover:bg-blue-100`).
5. THE Page_Header component SHALL render an `<h1>` with class `text-2xl font-bold text-gray-900`
   and an optional flex container for action buttons on every page that uses it.

---

### Requirement 6: PDF Report Template

**User Story:** As a shop manager, I want the income statement PDF template to be a standalone
print-ready document, so that it can be rendered by a PDF library without sidebar or navigation
chrome.

#### Acceptance Criteria

1. THE PDF_Template SHALL NOT extend Base_Template.
2. THE PDF_Template SHALL be a complete standalone HTML document with its own `<html>`, `<head>`,
   and `<body>` tags.
3. THE PDF_Template SHALL use print-safe inline CSS or minimal Tailwind classes and SHALL NOT
   include the dark Sidebar, top header bar, or any interactive JavaScript.
4. THE PDF_Template SHALL NOT contain any Bootstrap_Class.

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a
system — essentially, a formal statement about what the system should do. Properties serve as the
bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Active navigation state is exclusive

*For any* page rendered by the System, exactly the Sidebar navigation entry whose app matches the
current `request.resolver_match.app_name` SHALL carry the active state classes, and all other
entries SHALL carry only inactive state classes.

**Validates: Requirement 2.3**

### Property 2: Flash message color mapping is exhaustive

*For any* Django message with a tag value of `success`, `error`, `warning`, or `info`, the
Base_Template SHALL render that message wrapped in the corresponding color class set, and no
message SHALL be rendered without a color class.

**Validates: Requirement 2.4**

### Property 3: No Bootstrap classes in any template

*For any* file in Template_Set, the rendered source SHALL contain zero occurrences of any
Bootstrap_Class pattern (`col-md-`, `d-flex`, `btn btn-`, `table table-`, `badge bg-`, `row g-`,
`container`).

**Validates: Requirements 3.3, 2.5**

### Property 4: All non-PDF templates extend base.html

*For any* template in Template_Set except PDF_Template, the template source SHALL begin with
`{% extends 'base.html' %}` and SHALL contain `{% block content %}`.

**Validates: Requirement 3.2**

### Property 5: Empty-state row present for empty querysets

*For any* list-view template rendered with an empty queryset, the Data_Table SHALL contain a row
with a `colspan` attribute spanning all columns and a non-empty descriptive text string.

**Validates: Requirement 3.6**

### Property 6: Tailwind widget attrs present on all form fields

*For any* form class in Form_Files and *for any* field on that form, the field's widget `attrs`
dictionary SHALL contain the key `class` with a value that includes
`mt-1 block w-full rounded-md border-gray-300`.

**Validates: Requirements 4.2, 4.3, 4.4**

### Property 7: Rendered form inputs carry Tailwind classes

*For any* form from Form_Files rendered by Django's template engine, every `<input>`, `<select>`,
and `<textarea>` element in the rendered HTML SHALL contain the class attribute value
`mt-1 block w-full rounded-md border-gray-300`.

**Validates: Requirement 4.5**

### Property 8: Badge color variants are exhaustive and exclusive

*For any* Badge rendered in Template_Set, the Badge element SHALL carry exactly one of the four
defined color variant class pairs and SHALL NOT carry any Bootstrap badge class.

**Validates: Requirement 5.3**

### Property 9: Button variants are exhaustive and exclusive

*For any* action Button rendered in Template_Set, the Button element SHALL carry exactly one of the
four defined variant class sets and SHALL NOT carry any Bootstrap button class.

**Validates: Requirement 5.4**
